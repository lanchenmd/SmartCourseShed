## Problem Statement

公立中小学的排课管理需求变化频繁，现有商业软件响应慢（改一次需求要等很久），学校管理人员被迫手工处理大量冲突检测工作。核心痛点不是"排课"本身，而是"快速适应需求变化"。

## Demand Evidence

- 商业伙伴接触过多所公立学校，有实际需求沟通
- "如果好用，他们愿意付钱"——有付费意愿但尚未锁定具体客户
- 现有软件（商业软件）改需求周期长，学校处于被动地位

## Status Quo

学校购买商业排课软件，但：
- 软件商改一次需求要等很久
- 软件不支持自动检测冲突，只能手工逐行检查
- 开学前调课场景：教师请假 + 换教室 + 班级调整 → 手工检查容易遗漏

典型用户：教务主任、年级长、班主任，负责全校或本年级的排课工作。

## Target User & Narrowest Wedge

**典型用户：** 某初中教务主任王老师，负责全校排课
- 开学前一周面临：物理李老师请假一周 + 初二3班换教室
- 现在：手工逐行检查冲突，可能漏掉
- 期望：一键检测冲突 + 自动提示 + 手动调整

**最小可行版本（MVP）：**
- 配置驱动排课核心（不是对话驱动）
- 能生成完整课表 + 实时冲突检测
- 日历拖拽界面，支持手动调整
- 硬约束（必须满足）+ 软约束（尽量满足）

**Why MVP first：** 需要先验证"排课能做好"，再叠加"自我进化"能力。

## Constraints

- 技术栈已选定：Next.js 14 + Taro 4 + Node.js + Hono + PostgreSQL + Redis + OR-Tools
- 6 Agent 体系：需求沟通 → 校对 → 产品设计 → 开发 → 测试 → 运维
- 三级权限：超级管理员 / 管理员 / 普通用户
- MVP 只做单租户系统，架构预留扩展性

## Premises

1. **P1（已确认）：** 核心竞争力是"自我进化能力"（对话驱动重构），不是"排课算法本身"——MVP 先做配置驱动，自我进化是后续阶段
2. **P2（已确认）：** MVP 先做"配置驱动排课 + 冲突检测"，Agent 进化能力Phase2-3 的目标
3. **P3（修订后）：** MVP 只做单租户系统，多租户和私有化部署在架构上预留扩展性，但不在 MVP 阶段实现

## Approaches Considered

### Approach A: 增量构建（RECOMMENDED）

| Phase | 交付内容 |
|-------|----------|
| Phase1 | 配置驱动排课核心：硬约束 + 软约束 + 冲突检测 + 日历拖拽 UI |
| Phase2 | 叠加 Agent 体系：需求沟通 Agent + 校对 Agent |
| Phase3 | 实现自动重构能力：对话驱动规则更新 |

**Pros:**
- 每期都能交付可用产品
- 降低技术风险
- 快速验证市场需求

**Cons:**
- Phase1 的排课核心必须有足够竞争力

### Approach B: 完整先行

- 第一期就实现完整的 Agent 体系 + 自我重构能力
- 优势：差异化最强
- 风险：周期长，Phase1 无法交付可用产品

## Recommended Approach

**选择 Approach A（增量构建）**

理由：
1. MVP 需要快速验证市场需求（有学校愿意付费）
2. 排课核心（配置驱动）是基础，Agent 进化能力是增量叠加
3. 用户已明确：先有"好用"的产品，再谈"自我进化"

**MVP 范围（Phase1 已精简）：**

| 模块 | 功能点 |
|------|--------|
| 排课核心 | 全量排课（Python + OR-Tools）、增量排课、手动排 + 自动填充 |
| 约束系统 | 硬约束（必须满足）、软约束（尽量满足，满意度评分） |

| 用户权限 | 两级角色（管理员 + 普通用户）+ JWT 双 Token |

**Phase1 砍掉，Phase2实现：**

- Excel 批量导入

- 微信小程序端

- RAG 智能问答

## MVP 术语表与关键定义

### 冲突检测定义

**三维冲突（硬约束）：**
同一时间槽内，以下任意两项同时成立即为冲突：
1. **教师冲突** — 同一教师被安排到两个不同班级
2. **教室冲突** — 同一教室被安排给两个不同班级（或超出容量）
3. **班级冲突** — 同一班级被安排两门不同课程

**软约束冲突（不影响生成，但影响满意度评分）：**
- 教师时间偏好（如"不希望排在上午第一节"）
- 课程连堂偏好（如"体育课不要连排"）
- 科目分布均匀性（如"语文不要集中在上午"）

### 满意度评分

量化方式：0-100 分
- 100 分：全部软约束满足
- 80-99 分：1-2 个软约束轻微违反
- 60-79 分：3-5 个软约束违反
- <60 分：超过 5 个软约束违反，课表质量不理想

> **Phase2 适用：** 满意度评分依赖 L2 软约束体系，Phase1 不实现。Phase1 不计算满意度评分。

### 可接受课表

满足以下全部条件：
1. 硬约束冲突 = 0
2. 每个班级每门课程课时数精确等于配置的数量（`==`，非 `≥`）
3. **Phase2 新增：** 满意度评分 ≥ 60（L2 软约束上线后适用）

### 排课模式

| 模式 | 触发条件 | 算法行为 |
|------|----------|----------|
| 全量排课 | 学期初从零生成 | OR-Tools 全量求解，30 秒超时，返回最优解 |
| 增量排课 | 已有课表，局部调整 | OR-Tools 局部求解（只重排受影响的时间槽） |
| 手动排 + 自动填充 | 管理员锁定关键课节 | 未锁定部分由算法自动填充 |

## 调课/代课/请假流程（Phase2 实现）

| 场景 | 流程 | 通知 |
|------|------|------|
| 调课（教师申请） | 教师提交 → 系统检测冲突 → 无冲突则自动生效 | 站内通知 |
| 代课（教师申请） | 教师提交 → 管理员指定代课教师 → 系统确认 → 生效 | 站内通知 |
| 请假（教师申请） | 教师提交 → 管理员审核 → 生效后标记请假时段 | 站内通知 + 48h 超时提醒 |

## Architecture Notes

### 技术栈

| 组件 | 选择 | 理由 |
|------|------|------|
| 前端 Web | Next.js 14 (App Router) + React 18 + Ant Design 5 | SSR + 成熟 UI 生态 |
| 前端小程序 | Taro 4 (React) | 复用 Web 组件，Phase2 再做 |
| 后端主服务 | Node.js + Hono (JavaScript/TypeScript) | 成熟生态、易于招人和排障 |
| 排课算法服务 | Python + OR-Tools（独立服务，HTTP/REST 通信） | OR-Tools 是 Python 库，需独立部署；HTTP/REST 简化 Phase1 |
| 数据库 | PostgreSQL 16 | 关系型数据 + RLS 预留；所有表含 school_id |
| 缓存 | Redis | Token 黑名单、乐观锁（课表写操作并发保护） |
| AI 抽象层 | 统一 provider 接口 | 隔离 LLM SDK，可替换（Phase2-3） |

### 部署架构（Phase1）

```
单一租户系统（Docker Compose）：
- Nginx（反向代理）
- Next.js 应用
- Node.js + Hono 后端 API
- Python + OR-Tools 排课算法服务
- PostgreSQL
- Redis
```

### 多租户预留

- 所有表预留 `school_id` 字段
- PostgreSQL RLS 策略预留（暂不启用）
- Agent 模块设计为 per-school 可扩展（微服务形态）

## MVP 差异化定位（Phase1）

**Phase1 主打差异化：** 极致的冲突检测体验 + 超快的排课速度

- 国内现有排课软件冲突检测普遍做得差（只能检测部分冲突）
- 我们的目标：100% 硬约束冲突检测，一个不漏
- 排课速度：30 秒内生成可接受课表（竞品通常需要 5 分钟+）

**Phase1 核心价值：** 让教务主任从"手工检查冲突 1-2 天"变成"一键生成，2 小时内完成"

## Open Questions

1. **MVP 完成后的定价策略？** 尚未确定，按校收费 vs 其他模式
2. **Agent 体系的自动化程度？** "自动作业"置信度阈值尚未定义
3. **具体学校客户？** 商业伙伴接触过，但未锁定具体学校和联系人

## Success Criteria

1. **可用性：** 能在 30 秒内为 20 班级 + 50 教师生成可接受的课表
   - **前提：** Phase1 第 1 周须完成 OR-Tools Benchmark，用真实数据结构测试求解时间
   - 若 30 秒内无法得到可接受解，须调整算法策略（近似求解 + 分步求解）
2. **冲突检测：** 100% 覆盖硬约束冲突，不漏报
3. **用户满意度：** 管理员能在 2 小时内完成开学前调课（原来可能需要 1-2 天）
4. **商业验证：** 有 3所以上学校愿意付费
