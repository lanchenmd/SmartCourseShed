# 中小学排课系统
## 多 Agent 协作开发框架

需求规划 · 技术架构 · Agent 体系

---

## 一、项目背景与核心挑战

本项目是一个面向中小学的排课管理系统，支持网页端（Web）与微信小程序端，两端须具备完整功能，架构设计须为将来可能扩展的原生 App 端（iOS/Android）预留接入层，避免形成平台耦合。暂不开发 App 端，但技术选型须确保届时可复用核心业务逻辑层。

核心挑战在于：每所学校、每个年级、每个班级的排课需求差异显著，不存在通用模板，因此是否"快速响应需求变化"是整个系统成功的关键前提，这要求核心业务模型和约束求解模块保持稳定，其它模块应适应新需求快速迭代。

---

## 二、用户角色与权限体系

系统设三级用户角色，权限严格隔离，不可跨级操作。

### 2.1 超级管理员（Super Administrator）

由学校指定，每所学校唯一（或极少数几位）。无人工介入时，超级管理员为 Super Admin User Agent，可根据规则自动作业。

**职责与权限：**

- 可手工，也可基于规则自动审核并验证教师的实名注册申请，通过后方可激活账号（见第三节）
- 可手工，也可基于规则自动对已激活用户分配或变更角色：指定某用户为“管理员”或保持“普通用户”身份
- 超级管理员本身不参与日常排课具体工作，其操作仅限于账号管理与权限分配模块

> *约束：超级管理员无法直接操作排课数据，不得越过管理员层直接录入或修改排课内容。*

### 2.2 管理员（Administrator）

通常为年级长、班主任或学校指定的负责教师，由超级管理员授权。无人工介入时，管理员为 Admin User Agent，可根据规则自动作业。

**职责与权限：**

- 在自己责任范围内（如「初二年级」或「初二一班」）负责以下数据的手工或基于规则的自动录入与维护：
  - 学期信息
  - 课程科目及单节课时长
  - 教室信息（含实验室类型、教具设备条件）
  - 任课教师名单
  - 班级及学生人数、教师人数等基础数据
  - 排课约束条件（含硬约束与软约束）
- 在责任范围内执行排课操作，查看并管理排课结果
- 审核教师提交的调课、代课、请假申请（见第九节）

**多管理员协作规则：**

同一年级或班级允许存在多位管理员（如初二年级长与初二一班班主任可同时为管理员），系统须明确各管理员的数据责任边界，防止数据互相覆盖。具体边界划定逻辑须在需求沟通阶段与用户逐校确认。

> *约束：管理员仅能操作其被授权范围内的数据，不得访问或修改其他年级 / 班级的内容。*

### 2.3 普通用户（General User）

即全体任课教师（含未被授权为管理员的教师），此角色须为真人，其日常作业不得由 Agent 代理执行。

**职责与权限：**

- 查看与自身相关的课表信息
- 向所属管理员提交以下信息（系统须提供结构化提交入口，而非仅依赖线下沟通）：
  - 本学期承担的教学科目及课时
  - 个人时间偏好或限制（如不可排某天上午等）
  - 调课、代课、请假申请（须说明原因，由管理员审核）
  - 其他特殊要求（须说明原因，由管理员审核）

**兼任规则：**

若年级长或班主任本人也承担任课工作，其账号同时具备普通用户的全部权限，即可在管理员视图与普通用户视图之间切换，两套权限并行生效，互不干扰。

## 三、注册与登录机制

### 3.1 实名注册流程

1. 任课教师使用真实姓名、工号（或学校指定的唯一标识符）、联系方式进行注册，系统须防止同一身份重复注册
2. 注册成功后，账号进入「待审核」状态，用户不可登录
3. 超级管理员在管理后台收到待审核通知，核实身份后执行「审核通过」或「拒绝」操作
4. 审核通过后，系统自动激活账号并通知用户（站内 + 微信订阅消息），用户方可正式登录使用排课系统
5. 被拒绝的注册须提供拒绝原因，用户可在修正后重新提交申请

### 3.2 登录要求

- 支持账号密码登录；建议同时支持手机验证码登录
- 微信小程序端可结合微信授权，但须绑定实名信息，不得绕过审核流程
- 登录态须在网页端与小程序端独立维护，不共享 Session，但核心用户数据（权限、课表等）须在两端保持实时一致

## 四、客户端与技术架构要求

### 4.1 当前交付范围

- 网页版（Web）：须支持 PC 浏览器，兼容主流浏览器（Chrome、Edge、Safari、Firefox 最新两个大版本）
- 微信小程序版：须与网页版功能完全对等，不得有功能缺失或降级

### 4.2 架构兼容性要求（面向未来 App 扩展）

架构须遵循前后端分离原则，具体要求如下：

- 后端统一以 RESTful API 对外提供服务，所有业务逻辑集中于服务端，客户端不承载核心逻辑
- 网页端与小程序端均作为独立客户端调用同一套后端接口，禁止为某一端单独实现业务逻辑分支
- 接口设计须与客户端平台无关（Platform-agnostic），为将来 App 端接入预留完整能力，届时无需重构后端
- 认证鉴权采用标准 Token 机制（如 JWT），不依赖平台特定的 Session 机制，确保 App 端可直接复用
- AI 模型调用须通过统一抽象层封装，禁止在业务代码中直接耦合特定模型或供应商的 SDK，确保模型可替换、可升级，且不影响上层业务逻辑

## 五、Agent 体系定义

系统由以下六个 Agent 组成，各司其职，按流水线顺序协作，校对 Agent 在全流程四个关键节点穿插介入。

### Agent 1：需求沟通 Agent（Requirement Elicitation Agent）

职责：与用户（学校管理人员）进行多轮结构化对话，挖掘、澄清并最终输出明确的需求规格文档。

**交互要求（强制）：**

- 本 Agent 必须与用户进行真实的多轮对话，禁止一次性假设需求后直接输出文档
- 每轮对话须引导用户回答以下维度的问题（可分批次）：
  - 学校基本信息（年级数、班级数、教师数）
  - 课程种类与周课时数
  - 特殊约束（教师兼课、教室容量、连堂要求、走班制等）
  - 优先级规则（哪些约束是硬约束，哪些是软偏好）
  - 多管理员场景下的数据责任边界划分方式
  - 调课 / 代课 / 请假的审批流程与时效要求
  - 数据导入方式（是否有历史课表、是否对接现有教务系统）
  - 通知触达偏好（站内通知 / 微信消息 / 两者兼用）
  - 输出格式需求（导出 Excel、打印版、班级视图、教师视图等）

**歧义处理规则：**

- 若某需求经两轮沟通后仍不明确，Agent 必须主动提出 2～3 个可选方案供用户选择，每个方案须说明方案内容、适用场景、潜在局限
- 禁止在需求模糊时直接进入下一阶段

输出物：结构化需求规格文档（JSON 或 Markdown），包含：硬约束列表、软约束列表、优先级排序、用户角色与权限说明、验收标准（Acceptance Criteria）。

### Agent 2：校对 Agent（Verification & Consistency Agent）

职责：作为全流程质量守门人，在四个关键节点介入，对各阶段产出物进行独立验证。本 Agent 不参与生产，只负责审查与反馈。

**节点 A：需求规格审查（需求沟通 Agent 输出后）**

- [ ] 需求是否覆盖所有用户提及的约束场景？
- [ ] 三级角色的权限边界是否无歧义地描述清楚？
- [ ] 多管理员协作场景下的数据边界是否已明确？
- [ ] 注册审核流程是否完整闭环（含拒绝与重申流程）？
- [ ] 硬约束与软约束的边界是否清晰？
- [ ] 验收标准是否可测量、可验证？
- [ ] 是否存在自相矛盾的约束条目？
- [ ] 是否存在技术上不可行的需求（如：要求同一教师同一时段在两个教室上课）？
- [ ] 调课 / 代课 / 请假流程是否已完整覆盖？
- [ ] 数据导入 / 导出需求是否已明确？

**节点 B：产品设计一致性审查（产品设计 Agent 输出后）**

- [ ] 设计稿覆盖的功能点是否与需求规格一一对应？
- [ ] 三级角色的操作入口与权限隔离是否在设计上得到体现（如超级管理员无排课入口）？
- [ ] 网页端与小程序端的功能是否对等，无缺失或降级？
- [ ] 架构设计是否满足前后端分离、接口平台无关的要求？
- [ ] 是否存在设计中新增但需求未提及的功能？（若有，需标注「设计扩展项」并回传用户确认）
- [ ] 调课 / 代课 / 请假的完整流程是否有对应设计页面？
- [ ] 导入 / 导出功能的交互设计是否已覆盖？
- [ ] 用户流程是否满足【学校管理员无需技术背景即可独立完成排课】的易用性要求？

**节点 C：代码功能审查（代码开发 Agent 输出后）**

- [ ] 核心排课逻辑是否实现了需求规格中的全部硬约束？
- [ ] 软约束是否作为可配置项实现，而非硬编码？
- [ ] 权限鉴权逻辑是否覆盖三级角色，是否存在越权访问漏洞？
- [ ] 注册审核流程的后端实现是否与需求规格一致？
- [ ] 调课 / 代课 / 请假的后端流程是否完整实现？
- [ ] 审计日志是否对所有写操作完整记录，Agent 操作是否标注来源？
- [ ] 后端接口是否与平台无关，网页端与小程序端调用的是同一套接口？
- [ ] AI 模型调用是否通过抽象层封装，无直接 SDK 耦合？
- [ ] 多租户数据隔离是否在数据库层强制执行（RLS 策略）？
- [ ] 边界情况处理（如课时数无法整除、教师请假等）是否有明确的降级策略？

**节点 D：测试覆盖审查（测试 Agent 输出后）**

- [ ] 测试用例是否覆盖需求规格中的每一条验收标准？
- [ ] 是否包含三级角色权限隔离的专项测试？
- [ ] 是否包含注册审核全流程的端到端测试？
- [ ] 是否包含调课 / 代课 / 请假全流程的端到端测试？
- [ ] 是否包含多管理员并发操作同一数据的冲突测试？
- [ ] 是否包含针对硬约束冲突的边界测试用例？
- [ ] 是否包含不同学校规模的压力测试场景？
- [ ] 是否包含数据导入 / 导出的功能测试？
- [ ] 网页端与小程序端是否均有对应的功能回归用例？

输出物：每次审查输出一份《审查报告》，格式为：通过项列表、问题项列表（含严重等级：阻断 / 重要 / 建议）、建议修改方向。

> *反馈机制：发现阻断级问题时，当前阶段必须暂停，相关 Agent 修正后重新提交审查，方可进入下一阶段。*

### Agent 3：产品设计 Agent（Product Design Agent）

职责：基于需求规格文档（通过校对 Agent 审查后），输出完整的产品设计方案。

**输出物：**

- 功能模块清单（Feature List），含三级角色各自可见 / 可操作的功能范围说明
- 核心页面线框图（可用文字描述或 Mermaid 图），网页端与小程序端分别呈现
- 用户注册与审核流程图
- 调课 / 代课 / 请假完整业务流程图
- 数据导入 / 导出交互设计说明
- 数据模型草案（核心实体关系，含用户角色模型）
- 异常场景处理方案（排课冲突时的用户提示策略）
- API 接口规范草案（需说明接口如何保持平台无关性）

### Agent 4：代码开发 Agent（Software Development Agent）

职责：基于通过审查的产品设计方案，实现排课系统的核心功能。

**技术要求：**

- 采用前后端分离架构，后端提供统一 API，网页端与小程序端共用同一套接口
- 认证鉴权使用 JWT（Access Token + Refresh Token 双 Token 机制），不依赖平台特定 Session
- 三级角色的权限控制须在服务端强制执行，客户端的 UI 隐藏仅作为辅助手段，不作为安全边界
- 排课约束逻辑须封装为独立模块，便于按学校定制
- 约束规则须支持配置化（优先通过配置文件或数据库参数驱动，而非硬编码）
- 须提供排课结果的冲突检测接口，返回冲突原因及相关实体
- 所有写操作须记录审计日志，Agent 自动执行的操作须标注来源为「Agent」
- AI 模型调用须通过统一抽象层封装，禁止直接耦合特定模型 SDK

输出物：可运行的核心代码、API 接口文档、数据库 Schema（含用户角色与权限表设计）。

### Agent 5：测试 Agent（QA Agent）

职责：基于需求规格文档和代码，设计并执行测试。

**测试策略要求：**

- 测试用例必须溯源至需求规格的验收标准，做到逐条覆盖
- 须包含以下测试类型：
  - 功能测试（每条硬约束）
  - 权限隔离测试（三级角色的访问控制边界）
  - 注册审核流程端到端测试
  - 调课 / 代课 / 请假全流程端到端测试
  - 多管理员并发操作冲突测试
  - 配置测试（软约束开关验证）
  - 跨端一致性测试（同一操作在网页端与小程序端结果须一致）
  - 数据导入 / 导出功能测试
  - 边界测试（极小 / 极大规模学校）
  - 回归测试（修复后验证）

输出物：测试报告（含用例列表、通过率、未通过项及复现步骤）。

### Agent 6：运维上线 Agent（DevOps & Release Agent）

职责：在所有阶段通过校对后，负责系统的部署、配置和上线。

**上线前置条件（强制 Gate）：**

- 校对 Agent 节点 C 和 D 的审查报告均无阻断级问题
- 测试 Agent 报告的核心功能用例全部通过
- 网页端与小程序端均完成冒烟测试

输出物：上线清单（Deployment Checklist）、回滚预案、监控配置。

## 六、全局协作规则

1. 顺序性：各 Agent 按 1→3→4→5→6 的顺序推进，Agent 2（校对）在节点 A/B/C/D 穿插介入
2. 阻断机制：校对 Agent 发现阻断级问题时，流程暂停，相关 Agent 修正并重新提交，不允许带问题向下传递
3. 需求基线：需求规格文档一经校对 Agent 审查通过，即成为全流程的唯一真相来源（Single Source of Truth），所有后续 Agent 的产出须对照此文档进行验证
4. 用户再确认：若校对 Agent 在节点 B 发现设计扩展项，须暂停并由需求沟通 Agent 向用户二次确认，经用户同意后更新需求基线
5. 平台一致性原则：任何功能若在一个端实现，须同步在另一端实现。如确有例外（如小程序端微信授权绑定），须在产品设计阶段显式标注并经校对 Agent 确认
6. Agent 人工介入边界：超级管理员 Agent 和管理员 Agent 的自动作业须产生完整审计日志；当 Agent 遇到置信度低的决策时，须主动挂起并通知人工介入，而非静默失败

## 七、需求迭代指令

本项目上线后，每当用户有新需求时，请从 Agent 1（需求沟通 Agent）开始执行。首先向用户发出第一轮需求调研问题，不要假设任何学校信息，等待用户回答后再继续。

## 八、技术架构与选型

### 8.1 前端技术栈

**Web 端：Next.js 14（App Router）+ React 18**

- 服务端渲染（SSR）保障首屏加载速度
- TypeScript 全量覆盖，减少运行时错误
- UI 组件库：Ant Design 5.x（日历、表格、表单能力完善，有对应的小程序版 antd-mini）
- 状态管理：Zustand（轻量，适合中型应用规模）
- 拖拽排课组件：FullCalendar（支持卡片拖拽、多视图切换、外部元素拖入）

**小程序端：Taro 4（React 模式）**

- 复用 Web 端 React 组件逻辑，减少重复开发
- 微信授权登录须与实名信息绑定，不得绕过审核
- 为将来 App 端预留：增加编译目标即可，无需重写业务逻辑

### 8.2 日历排课交互设计（必须实现）

日历视图是管理员和教师最高频使用的界面，须满足：

**功能要求：**

- 支持周视图 / 月视图自由切换
- 课程卡片支持跨时间槽拖拽（换节）、跨日拖拽（换天），拖拽落点提供吸附动画
- 待排课程列表支持从边栏直接拖入日历时间格
- 冲突的课程卡片自动红框高亮，悬停提示冲突原因

**日期信息叠加层（双层信息显示）：**

- 每个日期列须同时显示：公历日期、农历日期（节气）、未来天气预报（调用和风天气或彩云天气 API）
- 法定节假日与调休日须以不同底色区分标注，春节前后等敏感排课窗口须视觉突出提示
- 农历计算使用本地库（如 lunar-javascript），无需联网，不增加 API 调用开销

**多管理员并发保护：**

- 同一时间槽被拖拽操作时，须通过 Redis 加乐观锁，防止两位管理员同时提交产生数据覆盖

### 8.3 后端技术栈

- 运行时：Bun（TypeScript）或 Python（FastAPI）
  - Bun 原生支持 TypeScript，无需 ts-node / tsx 等转译中间层；内置测试运行器（bun test）、包管理器（bun install）与打包器，减少工具链依赖；与 Node.js 高度兼容
  - 推荐 Web 框架：Elysia（Bun 原生，性能优先）或 Hono（轻量，兼容 Bun / Edge）
  - Docker 基础镜像：oven/bun 官方镜像
- API 规范：RESTful，版本化路径（如 `/api/v1/...`），所有端共用同一套接口
- 认证：JWT（Access Token + Refresh Token 双 Token 机制），Token 与平台无关，App 端将来可直接复用
- 实时通知（如排课更新推送）：WebSocket 或 Server-Sent Events（SSE）

### 8.4 数据库选型

**关系型数据库：PostgreSQL 16（主库，必选）**

排课数据高度结构化（课程、教师、教室、时间槽、约束规则之间有大量外键关联），关系型数据库是核心存储，不可替代。选择 PostgreSQL 的理由：

- SQLite 不支持多用户并发写入，不适合 Web 应用
- PostgreSQL 的 JSONB 字段可存储各校差异化的约束规则，无需为每所学校改表结构
- pgvector 扩展允许 PostgreSQL 同时承担向量存储职责（初期无需单独部署向量数据库）
- 窗口函数对排课冲突检测查询效率更高
- 完善的审计日志与历史版本支持
- 行级安全策略（Row-Level Security, RLS）可在数据库层强制多租户隔离，防止应用层漏洞导致跨校数据泄露

**多租户数据隔离策略：**

- 所有业务表加 school_id 字段，启用 PostgreSQL RLS 策略，在数据库层强制隔离
- 超级管理员账号须绑定 school_id，禁止跨校查询
- 向量检索须强制携带 school_id 过滤条件，禁止跨校检索

**向量数据库：pgvector（起步）→ Qdrant（规模扩展）**

- 起步阶段（单校或少量学校）：启用 PostgreSQL 的 pgvector 扩展，向量数据与关系数据存于同一实例
- 扩展阶段（多校接入、文档量破万）：将向量存储迁移至独立的 Qdrant 实例；由于 LangChain 封装了向量库接口，切换成本极低，仅更换存储后端配置

**缓存与消息：Redis**

- JWT Token 黑名单（登出失效）
- 排课时间槽乐观锁（多管理员并发保护）
- 日历天气数据缓存（减少第三方 API 调用频次）
- 实时通知消息队列

### 8.5 RAG 智能问答系统

使用场景：回答教师关于本校排课规章制度的查询，例如「调课需要提前几天申请」「实验室预约流程是什么」。文档来源为学校上传的内部制度文件（PDF / Word）。

**技术链路：**

1. 文档入库：上传的制度文件经解析后切分为语义段落，调用 Embedding 模型生成向量并存入 pgvector / Qdrant
2. 检索：用户提问后，对问题生成向量，在向量库中检索最相关的 3～5 个段落（强制携带学校 ID 过滤，确保各校数据隔离）
3. 生成：将检索到的段落 + 用户问题组合成 Prompt，通过 AI 模型抽象层调用 LLM 生成回答，回答中须标注原文出处段落，供用户核对
4. 框架：LangChain（Python）或 LlamaIndex

**Embedding 模型选型：**

- 优先：text2vec-base-chinese（开源，本地部署，零 API 费用）
- 备选：智谱 AI Embedding API（质量更高，有免费额度，适合对精度要求较高的场景）

### 8.6 第三方依赖

| 服务 | 用途 | 备注 |
|------|------|------|
| 和风天气 API | 日历天气显示 | 每日缓存，减少调用次数，可选 |
| lunar-javascript | 农历 / 节气计算 | 本地库，无需联网，无 API 费用，必须 |
| LLM API（Claude 等） | RAG 回答生成 | 按 Token 计费，须限流；须通过抽象层调用，禁止直接耦合 SDK，可选（RAG 模块启用时必须） |
| 微信开放平台 | 小程序授权登录 | 须绑定实名信息，必须（小程序端） |
| 微信订阅消息 | 课表变更实时通知 | 触达率高，须用户授权，推荐 |
| Excel 导入导出 | 批量数据导入 / 课表导出 | 使用 xlsx / exceljs 本地处理，必须 |

### 8.7 部署架构

初期推荐单机 Docker Compose 部署（适合单校或小规模 SaaS），服务组成：

- Nginx（反向代理）
- Next.js 应用
- Bun 后端 API 服务（基础镜像：oven/bun）
- PostgreSQL
- Redis
- Qdrant（可选，规模扩展阶段启用）

多校 SaaS 阶段迁移至容器编排（K8s 或托管服务），数据库升级为云托管实例（如 RDS）。

### 8.8 排课算法与求解策略

**算法选型：**

- 基于约束满足问题（CSP）的求解方式
- 推荐：Google OR-Tools（开源，Python / Java，工业级，支持大规模学校）
- 备选：自定义回溯 + 启发式算法（灵活，适合初期快速验证）

**求解模式：**

- 全量排课：学期初从零生成完整课表
- 增量排课：在已有课表基础上调整部分课节，不影响已锁定课节
- 手动排 + 自动填充：管理员先锁定关键课节，其余由算法自动填充

**冲突优先级：**

- 硬约束冲突：必须报错阻断，禁止生成包含硬约束冲突的课表
- 软约束冲突：给出冲突评分（满意度分数），由管理员决定是否接受当前方案

**求解超时策略：**

- 设置最大求解时间（推荐 30 秒），超时后返回当前最优解
- 标注未满足的软约束清单，供管理员手动调整

## 九、业务流程补充

### 9.1 调课申请流程

1. 教师发起申请：指定原课节 + 目标时间 / 教室
2. 系统自动检测目标时段冲突（教师、教室、班级三维冲突检测）
3. 管理员收到待审核通知，支持批量审核
4. 审核通过后课表自动更新，双方教师及受影响班级管理员收到通知
5. 审核不通过须填写原因，教师可修改后重新提交

> *调课申请须在变更日期前至少 N 天提交（N 由学校在系统配置中设定）。*

### 9.2 代课申请流程

1. 原教师发起代课申请，指定课节并可推荐代课教师（可选）
2. 管理员指定或确认代课教师
3. 代课教师在系统中确认接受
4. 仅本次生效，不影响原周期性课表；双方教师收到确认通知

### 9.3 教师请假流程

1. 教师提交请假申请，注明起止时间和原因
2. 管理员审核并批准
3. 系统自动标记该教师请假时段，检测受影响课节并通知管理员处理
4. 管理员为受影响课节安排代课或取消，更新后通知相关教师和班级

### 9.4 数据导入 / 导出规范

**导入支持：**

- 支持从 Excel 模板批量导入：教师名单、课程列表、班级信息、历史课表
- 系统提供标准导入模板下载，字段说明清晰，含示例数据
- 导入前须进行数据校验，错误行明确标注原因，不中断整体导入
- 为将来对接主流教务系统（正方、青果等）预留外部系统 ID 字段

**导出支持：**

- 导出格式：Excel（班级课表、教师课表、教室使用表）、PDF 打印版
- 图片分享版（适合小程序内转发给家长或学生）
- 课表变更后支持增量导出（仅导出本周改动部分）

## 十一、审计日志与操作追溯

- 所有写操作（排课、调课、权限变更、审核操作）须记录：操作人、操作时间、操作前值、操作后值
- Agent 自动执行的操作须标注来源为「Agent」，区别于人工操作，便于事后审查
- 审计日志对管理员可查（本责任范围内），超级管理员可查全校日志
- 日志保留周期：建议按学年归档，至少保留两个完整学年
- 提供日志导出功能（Excel / CSV），支持按时间范围、操作类型、操作人过滤
