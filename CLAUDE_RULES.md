# CLAUDE_RULES.md — 中小学排课系统 项目工作规则（Superpowers + gstack）

本文件是 Claude Code 在本项目中的**完整行为规范**，与根目录 CLAUDE.md 具有**同等效力**。
所有规则在整个开发周期内持续生效，无需每次提醒。

## 一、强制工作流（Superpowers）

### 1.1 开始任何新功能前
必须先完成 brainstorming，禁止直接开始写代码
brainstorming 结束前，不得创建任何业务代码文件
若需求描述不足以澄清设计方向，必须主动提问，
不得自行假设后推进

### 1.2 实现阶段
所有功能模块强制 TDD：先写失败测试 → 确认失败
→ 写最小实现 → 确认通过 → 提交
禁止在测试通过前进行下一个任务
每个任务完成后必须 commit，commit message
须说明实现了哪条验收标准

### 1.3 模块完成后
每个功能模块完成后，执行
verification-before-completion，
对照计划逐条确认后方可关闭该任务

### 1.4 遇到以下情况，必须暂停并通知我
发现需求文档与技术实现存在矛盾
某个硬约束在当前技术方案下无法满足
连续三次修复后同一问题仍未解决
对架构层面的方向产生不确定性

## 二、角色切换规则（gstack）

以下规则定义了何时必须主动切换到对应的 gstack 角色。
切换后须以该角色的认知模式完成检查，输出检查结论，
再切换回开发模式继续推进。

### 2.1 /office-hours — 需求澄清角色
**触发条件（满足任意一条即触发）：**项目启动，开始处理需求文档时（必须执行一次）
准备开始一个新的功能模块，但需求描述少于 3 个
明确的验收标准
某功能经过两轮讨论后方向仍不清晰

**执行方式：**
以 YC office hours 模式，用追问逼清楚以下问题：
这个功能解决谁的什么问题？边界在哪里？
成功的标准是什么？---

### 2.2 /plan-ceo-review — 方案决策角色
**触发条件（满足任意一条即触发）：**数据库 Schema 设计完成后
API 接口规范草案完成后
排课算法方案确定前
任何涉及"是否要做这个功能"的决策点

**执行方式：**
以产品策略视角审查方案，重点挑战：范围是否过度设计？
优先级排序是否合理？
这个决策将来是否难以推翻？

---

### 2.3 /review — 工程质量审查角色
**触发条件（每个模块完成后必须执行）：**排课核心约束求解模块完成后
用户权限与鉴权模块完成后
调课 / 代课 / 请假业务流程完成后
任何超过 200 行的新增代码提交前

**执行方式：**
以 staff engineer 视角审查，重点关注：生产环境下会出错的边界条件
性能瓶颈（尤其是数据库查询）
代码可维护性和模块边界清晰度

---

### 2.4 /cso — 安全审计角色
**触发条件（强制，不可跳过）：**JWT 认证 / 刷新 Token 机制实现完成后
三级角色权限控制模块完成后
多租户数据隔离（RLS 策略）配置完成后
任何涉及用户输入直接写入数据库的模块完成后

**执行方式：**
运行 OWASP Top 10 + STRIDE 威胁建模，
重点检查：越权访问、跨校数据泄露、SQL 注入、
Token 伪造。输出风险清单，阻断级风险修复后
才能继续推进。---

### 2.5 /qa — 功能验收角色
**触发条件：**每个完整功能模块提交后，进行端到端验收
发现 /review 或 /cso 中的问题被修复后，
回归验证

**执行方式：**
打开真实浏览器，按验收标准逐条操作验证，
不得仅凭代码阅读判断功能是否正确。---

### 2.6 /ship — 上线决策角色
**触发条件：**所有模块通过 /qa 验收后，准备部署前

**前置门禁（必须全部满足才能执行 /ship）：**/cso 无阻断级安全风险
/qa 全部验收标准通过
所有模块完成 verification-before-completion
回滚方案已准备好

## 三、Subagent 使用规则

### 3.1 必须使用 subagent 的场景
同时实现两个以上互相独立的模块
→ 使用 dispatching-parallel-agents
单个模块实现任务超过 15 个步骤
→ 使用 subagent-driven-development，
  主对话只接收最终结果和审查报告
/review 或 /cso 的审查任务
→ 派独立 subagent 执行，避免污染实现上下文

### 3.2 禁止使用 subagent 的场景
任务依赖刚才对话中澄清的业务规则
需要来回确认、方向未定的探索性任务
简单的单文件修改或 bug 修复

### 3.3 Subagent 汇报格式要求
每个 subagent 完成后须输出：完成了什么（对照计划逐条）
未完成或偏差项（及原因）
需要主对话决策的遗留问题

## 四、本项目特殊约定

### 排课核心模块保护
src/scheduler/ 目录下的约束求解逻辑，
任何修改前必须先执行 /review
禁止在业务代码中硬编码约束规则，
所有约束必须通过配置驱动

### AI 模型调用保护  
禁止在任何业务模块中直接 import 或调用
特定 LLM 的 SDK
所有模型调用必须经过 src/ai/provider.ts
抽象层，违反此规则的代码不得提交

### 数据隔离红线
所有数据库查询必须携带 school_id 过滤条件
任何缺少 school_id 过滤的查询，视为阻断级问题，
/cso 审查时必须拦截

### 多管理员并发保护
涉及课表写操作的接口，必须有 Redis 乐观锁，
/review 时重点检查此项
